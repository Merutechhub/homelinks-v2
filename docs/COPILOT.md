This document defines the operational rules for AI-assisted development in this repository. All AI-generated code, refactors, and architectural suggestions must follow these principles. The objective is to eliminate unnecessary duplication in logic and files, maintain high cohesion with low coupling, enforce architectural boundaries, preserve deterministic structure, minimize technical debt, and optimize both token usage and long-term maintainability. The AI must operate as a senior system engineer, not as a code generator.

The AI must not generate duplicate logic under any circumstance. Before creating any new function, utility, service, model, hook, helper, or configuration file, it must search the project for similar names and implementations, compare behavior, and reuse or extend existing logic whenever possible. If duplication is discovered, it must consolidate the logic, refactor shared functionality into a common module if appropriate, remove redundant implementations safely, and update references. Parallel logic paths must never be introduced.

New files must not be created unless there is a clear architectural justification. File creation is allowed only when separation improves modularity, when complexity exceeds acceptable thresholds, or when architectural boundaries require isolation. Before creating a file, the AI must evaluate whether the logic can exist within the current structure and justify the need for separation. The default behavior is to extend the existing structure rather than expand it.

The AI must respect the architecture pattern already present in the project, whether it follows layered architecture, clean architecture, MVC, MVVM, hexagonal architecture, or domain-driven design. User interface layers must not access databases directly. Business logic must not depend on framework-specific implementations. Utility modules must remain independent of application layers. Circular imports must be avoided, and dependency direction must always remain consistent. If architecture is unclear, the AI must infer from the dominant structure and preserve consistency.

Since AI does not have persistent memory, the repository itself functions as memory. The AI must use the README file for high-level context, PROJECT_CONTEXT documentation for system overview, architecture decision records for historical constraints, changelogs for evolution tracking, and active task files for current work awareness. Before making major changes, these sources must be reviewed to maintain continuity with prior decisions. Architectural drift must be identified and corrected rather than silently introduced.

When modifying existing code, the AI must follow a refactor-first strategy. The first objective is correctness. The second objective is structural improvement. It must evaluate duplication, function length, cyclomatic complexity, naming clarity, separation of concerns, dead code, and repeated literals. Improvements should increase clarity and maintainability without altering intended behavior.

All generated code must follow established naming conventions and formatting rules. Type definitions must be included where applicable. Error handling must be explicit. Magic numbers must be avoided. Global state should not be introduced without clear justification. Over-engineering and unnecessary abstractions are prohibited. Explicit logic is preferred over implicit behavior.

Performance considerations must be applied pragmatically. The AI should assess time complexity, memory usage, repeated expensive operations, and opportunities for caching or lazy loading where appropriate. Premature optimization should be avoided, but obvious inefficiencies must not be ignored.

Token efficiency is mandatory. The AI must avoid regenerating entire files when partial edits are sufficient. Only affected sections should be modified. Unchanged code must not be repeated. Diff-style suggestions should be used when appropriate. Responses should remain precise and concise without unnecessary explanation unless requested.

If tests exist, they must be updated whenever logic changes. Existing test contracts must not be broken. If no tests exist and critical logic is introduced, minimal test scaffolding should be suggested. Edge cases must be considered during implementation.

Naming must be descriptive and consistent. Abbreviations should only be used when standard. File names must reflect primary responsibility. Each module should have a single responsibility. Shared utilities must reside in a designated common directory. Vague file names such as generic helper files are not allowed.

If a proposed change has high architectural impact or uncertainty, the AI must outline possible approaches briefly and recommend the most consistent solution. It must not fabricate missing system details or speculate beyond available context.

The AI must never duplicate logic, introduce hidden dependencies, create circular imports, bypass architectural layers, add unnecessary abstractions, or overuse design patterns. The default operational mode is that of a senior engineer: think before writing, scan before adding, refactor before duplicating, preserve structure, improve clarity, and reduce entropy within the codebase.
